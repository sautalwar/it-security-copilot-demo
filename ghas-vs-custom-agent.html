Section 1

The Reality Check

Why GHAS alone isn't enough — and why a custom agent alone isn't either.

"Customers often ask — why do I need a custom agent if I have GHAS? Great question. GHAS is incredibly powerful for code-level and supply-chain security. But enterprise security goes far beyond source code. You have networks, infrastructure, compliance, VPNs, firewalls, SIEM — GHAS wasn't designed for those. That's where the custom agent comes in. Let me show you exactly where the line is."

GitHub Advanced Security (GHAS) is best-in-class for code security and supply chain security. It catches SQL injection, XSS, leaked secrets, and vulnerable dependencies at the source — before code ever reaches production.

But enterprise IT security spans far more than source code. Network monitoring, infrastructure auditing, compliance frameworks, and incident response live outside the codebase — in DNS logs, firewall rules, VPN configs, and SIEM platforms. GHAS wasn't designed for those domains.

The answer isn't GHAS or a custom agent. It's GHAS + a custom agent, each covering what the other can't.

How the Custom Agent Works

Prompt files as a version-controlled knowledge base, evaluated by a GitHub Actions workflow.

"This is the magic — the agent's intelligence comes from these prompt files. They're written in plain English mixed with specific thresholds and patterns. Your security team maintains them like they maintain runbooks. When a new threat emerges, you update the prompt file, push it, and the agent immediately knows about it. No redeployment, no new tooling — just a markdown file update."

🔑 Key Concept: Prompt Files as Knowledge Base

The custom agent uses markdown "prompt files" stored in .github/security-agent/ that contain security rules and standards, detection patterns (regex, thresholds, indicators), remediation templates, and compliance framework mappings. These files are version-controlled, reviewable, and updatable — security-as-code.

📝 Version-Controlled

Every rule change is tracked in Git history. Audit who changed what, when, and why via PR reviews.

👀 Reviewable

Security rules go through the same PR review process as code. Require approval from security team leads.

⚡ Instantly Updatable

New threat? Update the markdown file. Push. The agent uses the new rules on its next run. No rebuild needed.

🔄 Composable

Mix and match prompt files per job. DNS job reads DNS rules, VPN job reads VPN rules. Clean separation of concerns.

# DNS Tunneling Detection Rules

## High-Entropy Subdomain Detection
- Threshold: Shannon entropy > 3.5 for subdomain labels
- Pattern: Subdomain length > 30 characters
- Pattern: Base64-like character distribution (A-Za-z0-9+/=)
- Alert: CRITICAL when > 10 high-entropy queries
  to same domain in 5 minutes

## Known-Bad Domain Indicators
- C2 patterns: Random subdomain + static apex domain
- DGA patterns: Algorithmically generated domain names
- Fast-flux: Multiple A records changing rapidly

## Rate Limiting Thresholds
- Alert: > 50 queries/minute from single source IP
- Alert: > 20 TXT queries/hour from single source IP
- Alert: > 100 unique subdomains/hour for same apex domain

## Approved DNS Resolvers
- 168.63.129.16 (Azure DNS)
- 10.0.0.4 (Internal DNS)
- Block: All other outbound DNS (port 53)

# VPN Cryptographic Standards

## Forbidden Algorithms (CRITICAL)
- DES, 3DES — block cipher too weak
- MD5 — hash collision attacks proven
- DH Group 1 (768-bit), DH Group 2 (1024-bit)
- SHA-1 for integrity

## Required Minimums
- Encryption: AES-256-GCM or AES-256-CBC
- Integrity: SHA-256 or SHA-384
- Key Exchange: DH Group 14 (2048-bit) minimum,
  DH Group 20 (ECDH 384-bit) preferred
- IKE version: IKEv2 required (IKEv1 deprecated)

## Certificate Requirements
- Minimum RSA key size: 2048-bit
- Preferred: ECDSA P-384
- Alert: Certificates expiring within 30 days
- CRITICAL: Certificates using SHA-1 signatures
Section 4

The Agent Workflow Implementation

A complete GitHub Actions workflow that covers ONLY the gaps GHAS doesn't — no duplication.

"Notice this workflow explicitly does NOT scan for secrets or code vulnerabilities — GHAS handles those. Each job here maps to a specific domain that GHAS can't see: DNS, firewalls, VPN, compliance, incident response. The prompt files feed each job its detection rules."

YAML 
name: "🤖 Custom Security Agent"

# Covers ONLY what GHAS doesn't:
# ✗ NOT secrets (GHAS Secret Scanning)
# ✗ NOT code vulns (GHAS CodeQL)
# ✗ NOT dependency CVEs (GHAS Dependabot)
# ✓ DNS monitoring, Firewall auditing, Network forensics
# ✓ IaC deep CIS checks, WAF/ZeroTrust validation
# ✓ VPN crypto audit, Policy enforcement
# ✓ Network compliance, Incident reporting

on:
  schedule:
    - cron: '0 */6 * * *'      # Every 6 hours
  workflow_dispatch:
    inputs:
      scope:
        description: 'Scan scope'
        type: choice
        options:
          - all
          - network
          - infrastructure
          - compliance
        default: 'all'

permissions:
  contents: read
  issues: write
  pull-requests: write

env:
  RULES_DIR: .github/security-agent
  SEVERITY_THRESHOLD: HIGH

jobs:
  # ─── Job 1: DNS Threat Monitor ───────────────
  dns-monitoring:
    name: "🌐 DNS Threat Monitor"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'network'
    steps:
      - uses: actions/checkout@v4

      - name: Load DNS security rules
        id: rules
        run: |
          echo "Loading prompt file..."
          cat ${{ env.RULES_DIR }}/dns-security-rules.md
          echo "rules_loaded=true" >> $GITHUB_OUTPUT

      - name: Analyze DNS logs for exfiltration
        run: |
          python scenarios/01-dns-alert/remediate.py \
            --rules ${{ env.RULES_DIR }}/dns-security-rules.md \
            --output dns-findings.json

      - name: Post findings to summary
        if: always()
        run: |
          echo "## 🌐 DNS Threat Monitor" >> $GITHUB_STEP_SUMMARY
          python scripts/format-findings.py dns-findings.json \
            >> $GITHUB_STEP_SUMMARY

  # ─── Job 2: KQL Query Generator ─────────────
  kql-siem:
    name: "🔍 KQL/SIEM Query Generator"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'network'
    steps:
      - uses: actions/checkout@v4
      - name: Generate detection queries
        run: |
          python scenarios/02-kql-siem/generate_queries.py \
            --rules ${{ env.RULES_DIR }}/siem-detection-rules.md

  # ─── Job 3: Firewall Audit ──────────────────
  firewall-audit:
    name: "🔥 Firewall Rule Audit"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'network'
    steps:
      - uses: actions/checkout@v4
      - name: Validate firewall rules
        run: |
          python scenarios/03-firewall/audit.py \
            --rules ${{ env.RULES_DIR }}/firewall-best-practices.md \
            --config configs/firewall-rules.json

  # ─── Job 4: Network Forensics ───────────────
  network-forensics:
    name: "🔬 Network Forensics"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'network'
    steps:
      - uses: actions/checkout@v4
      - name: Extract IOCs and build timeline
        run: |
          python scenarios/04-forensics/analyze.py \
            --rules ${{ env.RULES_DIR }}/network-forensics.md

  # ─── Job 5: (SKIP — GHAS Secret Scanning) ───
  # Secret scanning handled by GHAS natively.
  # Enable in: Settings → Code security → Secret scanning

  # ─── Job 6: IaC CIS Benchmark Scan ─────────
  iac-hardening:
    name: "🏗️ IaC CIS Benchmark Scan"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'infrastructure'
    steps:
      - uses: actions/checkout@v4
      - name: Deep CIS scan for Bicep/Terraform
        run: |
          python scenarios/06-iac-harden/scan.py \
            --rules ${{ env.RULES_DIR }}/iac-cis-benchmarks.md \
            --glob "**/*.{bicep,tf,json}"

  # ─── Job 7: WAF & Zero Trust Validation ─────
  waf-zerotrust:
    name: "🛡️ WAF & Zero Trust"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'infrastructure'
    steps:
      - uses: actions/checkout@v4
      - name: Validate WAF and segmentation
        run: |
          python scenarios/07-waf-zerotrust/validate.py \
            --rules ${{ env.RULES_DIR }}/waf-zerotrust-rules.md

  # ─── Job 8: VPN Crypto Audit ────────────────
  vpn-audit:
    name: "🔒 VPN Crypto Audit"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'infrastructure'
    steps:
      - uses: actions/checkout@v4
      - name: Audit VPN configurations
        run: |
          python scenarios/08-vpn-audit/audit.py \
            --rules ${{ env.RULES_DIR }}/vpn-crypto-standards.md

  # ─── Job 9: Policy-as-Code Enforcement ──────
  policy-enforcement:
    name: "📜 Policy-as-Code"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'compliance'
    steps:
      - uses: actions/checkout@v4
      - name: Evaluate OPA/Rego policies
        run: |
          python scenarios/09-policy-code/evaluate.py \
            --rules ${{ env.RULES_DIR }}/policy-definitions.md

  # ─── Job 10: Network Compliance ─────────────
  network-compliance:
    name: "📋 Network Compliance Monitor"
    runs-on: ubuntu-latest
    if: >-
      inputs.scope == 'all' ||
      inputs.scope == 'compliance'
    steps:
      - uses: actions/checkout@v4
      - name: CIS/NIST/PCI benchmark check
        run: |
          python scenarios/10-compliance/monitor.py \
            --rules ${{ env.RULES_DIR }}/compliance-frameworks.md

  # ─── Job 11: (SKIP — GHAS CodeQL) ──────────
  # Vulnerability detection handled by GHAS CodeQL.
  # Enable in: Settings → Code security → Code scanning

  # ─── Job 12: Incident Report Generator ──────
  incident-report:
    name: "📊 Incident Report"
    needs:
      - dns-monitoring
      - firewall-audit
      - network-forensics
      - iac-hardening
      - vpn-audit
      - network-compliance
    runs-on: ubuntu-latest
    if: always()
    steps:
      - uses: actions/checkout@v4

      - name: Correlate findings & generate report
        run: |
          python scenarios/12-incident-report/generate.py \
            --rules ${{ env.RULES_DIR }}/incident-report-template.md \
            --mitre-mapping ${{ env.RULES_DIR }}/mitre-attack-map.md

      - name: Create issue for critical findings
        if: env.HAS_CRITICAL == 'true'
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const report = fs.readFileSync('incident-report.md','utf8');
            await github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: '🚨 Security Agent: Critical Findings',
              body: report,
              labels: ['security','critical','automated']
            });

📌 Notice: Jobs 5 and 11 are intentionally skipped

Secret scanning (Job 5) and code vulnerability detection (Job 11) are omitted because GHAS handles them natively. The custom agent workflow covers only the 8 domains GHAS doesn't reach: DNS, KQL/SIEM, firewalls, forensics, IaC deep scanning, WAF/Zero Trust, VPN, and compliance.

Common Customer Objections & Responses

❓ "We already have a SIEM — why do we need this?"

"Great — the agent feeds your SIEM. It auto-generates KQL queries and correlates events so your SOC analysts don't have to write detection rules from scratch. It makes your SIEM investment more effective."

❓ "Can't we just use Defender for Cloud?"

"Defender for Cloud is great for Azure posture. The custom agent complements it by covering on-prem infrastructure, multi-cloud configs, and custom compliance frameworks specific to your org. Plus it integrates directly into the developer workflow via GitHub."

❓ "How do we maintain the prompt files?"

"Same way you maintain runbooks — but better. They're in Git, so every change is reviewed, approved, and auditable. Your security team updates them when new threats emerge or compliance requirements change. No specialized tooling needed — just markdown."

❓ "What about false positives?"

"The detection thresholds are defined in the prompt files and tunable. Start conservative, then tighten based on your environment. Each finding includes confidence scores and context so analysts can triage quickly."
