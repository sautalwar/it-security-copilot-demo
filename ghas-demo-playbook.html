Dependency Graph — Deep Dive

What it does: The Dependency Graph automatically parses your manifest files (package.json, requirements.txt, pom.xml, go.mod, Gemfile, etc.) and builds a complete map of every direct and transitive dependency your project uses. It understands the full tree — not just what you declared, but what your dependencies depend on.

Out-of-the-box open source detection: Yes — it detects open source libraries automatically for 20+ ecosystems: npm, PyPI, Maven, NuGet, RubyGems, Go modules, Cargo (Rust), Composer (PHP), pub (Dart), Hex (Elixir), and more. No configuration needed — just push a manifest file and the graph builds itself.

How it differentiates from competition:

  • GitHub Advisory Database: World's largest vulnerability database with community-reviewed CVEs. Snyk and Sonatype use their own proprietary databases — GitHub's is open, transparent, and curated by the community + GitHub security researchers.
  • Transitive dependency resolution: GitHub resolves the full dependency tree, including transitive deps. Competitors like Snyk do this too, but GitHub does it natively without any CLI tool or agent installation.
  • SBOM export: One-click Software Bill of Materials in SPDX format — critical for compliance (Executive Order 14028 requires SBOMs for federal software). Most competitors charge extra for SBOM generation.
  • Submission API: Unlike competitors, GitHub lets build systems and package managers submit dependency snapshots via the Dependency Submission API, covering ecosystems that don't have manifest files (e.g., Bazel, Make, custom build systems).
  • Zero friction: No agent to install, no CLI to run, no config file to write. Just enable it. Snyk requires a CLI tool; Sonatype requires Nexus Lifecycle; Checkmarx SCA requires an agent. GitHub does it with zero setup.

Custom/private libraries: For internal packages hosted in GitHub Packages, Azure Artifacts, or private registries:

  • Dependency Submission API: Use the POST /repos/{owner}/{repo}/dependency-graph/snapshots API to submit custom dependency data. Your CI/CD pipeline can call this after resolving dependencies, including private ones.
  • GitHub Actions: Use the @github/dependency-submission-toolkit to build custom submitters that parse your proprietary build outputs and submit them to the graph.
  • Manifest files: If your private packages are listed in standard manifest files (e.g., private npm packages in package.json with a scoped registry), the graph detects them automatically.
  • Custom advisory databases: You can create private security advisories for your internal libraries in your org, so Dependabot alerts fire for your own packages too.

Tip: If customer asks "does it work with our internal Maven repo?" — answer is yes, as long as the pom.xml references the dependency. The graph parses the manifest, not the registry.

💡 Deep Dive: "Does it work with our internal Maven repo?"

This is one of the most common enterprise questions. Here's the full picture:

How Dependency Graph actually works:

The Dependency Graph does NOT connect to any registry. It reads your manifest files (pom.xml, package.json, requirements.txt, etc.) that are checked into your GitHub repo. It parses the XML/JSON/text to extract package names and version numbers. That's it — it's a file parser, not a registry client.

So for a pom.xml like this:

<dependencies>
  <!-- Public dependency from Maven Central -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>3.2.0</version>
  </dependency>

  <!-- Internal dependency from your private Nexus/Artifactory -->
  <dependency>
    <groupId>com.contoso.internal</groupId>
    <artifactId>auth-library</artifactId>
    <version>2.4.1</version>
  </dependency>
</dependencies>

The Dependency Graph will show both dependencies — the public Spring Boot one AND your internal com.contoso.internal:auth-library. It doesn't care where they come from. It just reads the pom.xml.

But here's the nuance — three different concerns:

Concern Public Packages Internal Packages
1. Visibility in Dependency Graph ✅ Automatic — parsed from manifest ✅ Automatic — parsed from manifest (same behavior)
2. CVE/Vulnerability Alerts ✅ Automatic — matched against GitHub Advisory Database (200K+ advisories) ⚠ Only if you create private security advisories for your internal packages. GitHub's advisory DB only knows about public open source packages.
3. Dependabot Auto-Update PRs ✅ Automatic — Dependabot resolves from Maven Central ⚠ Needs configuration — Dependabot must be told how to reach your private registry (Nexus, Artifactory, Azure Artifacts)
4. Transitive Dependencies ✅ Resolved from lock files or build output ⚠ Needs Dependency Submission API or self-hosted runner that can access the private registry to resolve the full tree

Configuring Dependabot for private Maven registries:

Add a .github/dependabot.yml file to your repo:

# .github/dependabot.yml
version: 2
registries:
  contoso-nexus:
    type: maven-repository
    url: https://nexus.contoso.internal/repository/maven-releases/
    username: ${{secrets.NEXUS_USERNAME}}
    password: ${{secrets.NEXUS_PASSWORD}}

updates:
  - package-ecosystem: "maven"
    directory: "/"
    schedule:
      interval: "weekly"
    registries:
      - contoso-nexus    # Tell Dependabot to check this registry too

Dependabot — Self-Hosted Runners & Azure Runners

What you see in Settings: There's a specific toggle for "Dependabot on self-hosted runners" (visible in the settings page). This is key for enterprise customers.

Why self-hosted matters for Dependabot:

  • Private registries: If your org uses private npm/Maven/NuGet registries behind a firewall, GitHub-hosted runners can't reach them. Self-hosted runners inside your network CAN — so Dependabot can resolve and update private dependencies.
  • Air-gapped environments: Some enterprises (finance, government, defense) have air-gapped networks. Self-hosted runners let Dependabot run inside those environments.
  • Custom CA certificates: If your private registry uses internal CA certs, self-hosted runners can be configured with those certs so Dependabot trusts the connection.
  • Performance: For monorepos with thousands of dependencies, self-hosted runners with more CPU/RAM can resolve faster than GitHub-hosted runners.

Azure-hosted runners: You can use Azure VMs as self-hosted runners (or GitHub's new Azure-hosted larger runners). Configure by:

  1. Provision an Azure VM (Ubuntu/Windows) or use Azure Virtual Machine Scale Sets for auto-scaling
  2. Install the GitHub Actions runner: ./config.sh --url https://github.com/ORG --token TOKEN
  3. Label it self-hosted, dependabot
  4. Enable "Dependabot on self-hosted runners" in org/repo settings
  5. Dependabot jobs now run on YOUR infrastructure with access to internal resources

Actions Dependabot can execute:

  • Security updates: Auto-creates PRs to bump vulnerable dependencies to the nearest patched version
  • Version updates: Keeps dependencies current (configurable via dependabot.yml — daily/weekly/monthly)
  • Grouped updates: Bundles multiple dependency updates into a single PR to reduce noise (new feature visible in settings)
  • Auto-merge: With branch protection + required checks, Dependabot PRs can auto-merge if CI passes — fully automated patching
  • Custom registries: Configure private registries in dependabot.yml with auth tokens stored as repo secrets
  • Ignore rules: Skip specific versions or dependencies you don't want updated
  • Reviewers/assignees: Automatically assign security team members to review Dependabot PRs

Competitive edge: Snyk requires a separate pipeline integration for auto-PRs. Mend (formerly WhiteSource) requires a bot installation. Dependabot is native to GitHub — zero additional tooling.

CodeQL — How It Works

CodeQL is not just pattern matching — it's a semantic code analysis engine. It compiles your source code into a relational database, then runs queries against that database to find vulnerabilities. Think of it as "SQL for your source code."

The process:

  1. Build: CodeQL compiles your code (or parses it for interpreted languages) into a database that captures the AST, data flow, control flow, and type information
  2. Analyze: It runs hundreds of pre-built security queries that trace data flow from "sources" (user input) to "sinks" (dangerous operations like SQL queries)
  3. Report: Findings appear as Code Scanning alerts with exact file/line, data flow path, severity, and CWE classification

Why CodeQL Beats the Competition

  • Data flow analysis, not regex: Competitors like SonarQube and Checkmarx SAST use pattern matching and taint analysis. CodeQL models your entire codebase as a queryable database. It finds vulnerabilities that span multiple files and function calls — things regex can never catch.
  • Open query language: CodeQL queries are open source (github/codeql repo). Your security team can write custom queries in QL. Checkmarx and Fortify queries are proprietary black boxes — you can't see what they check or customize them deeply.
  • Community-driven: 500+ community-contributed queries. Security researchers worldwide contribute detection logic. Proprietary SAST tools depend on a single vendor's research team.
  • Copilot Autofix: This is the killer differentiator. When CodeQL finds a vulnerability, Copilot Autofix generates the fix code — parameterized queries, input validation, output encoding. No other SAST tool can auto-generate fixes with AI. SonarQube shows you the problem; GitHub shows you the problem AND writes the fix.
  • Zero configuration (Default Setup): Enable with one click. No build configuration, no exclusion rules, no agent installation. SonarQube requires a server deployment. Checkmarx requires project onboarding. Fortify requires a license server. CodeQL just works.
  • PR integration: Findings appear directly in the PR diff — developers see them in context, not in a separate tool. This is shift-left done right.
  • Free for public repos: CodeQL is free for all public repositories. Competitors charge per-user or per-scan.

Supported Languages

C/C++, C#, Go, Java/Kotlin, JavaScript/TypeScript, Python, Ruby, Swift. Covers ~85% of enterprise codebases. Competitor comparison:

  • SonarQube: 30+ languages but shallower analysis (mostly pattern-based)
  • Checkmarx: 25+ languages with good Java/C# depth but weaker on JS/Python
  • Fortify: 25+ languages but slow scans and expensive licensing
  • CodeQL: Fewer languages but DEEPER analysis on each — quality over quantity

Self-Hosted Runners for CodeQL

Why self-hosted matters for CodeQL:

  • Large codebases: CodeQL analysis of a 1M+ line Java codebase can take 30-60 minutes and needs 8GB+ RAM. GitHub-hosted runners have 7GB RAM. Self-hosted runners with 16-32GB handle large repos much better.
  • Build dependencies: Compiled languages (C/C++, Java, C#) need the full build toolchain. If your build requires proprietary SDKs, internal libraries, or licensed compilers, self-hosted runners can have those pre-installed.
  • Air-gapped/regulated: For customers who can't send code to GitHub-hosted runners (FedRAMP, ITAR, financial regulations), self-hosted runners keep everything on-premises.
  • Faster iteration: Self-hosted runners with warm caches and pre-built CodeQL databases analyze faster on subsequent runs.

Azure-hosted CodeQL runners:

  1. Use GitHub's larger runners (Azure-backed) with 8-64 cores and up to 256GB RAM for enterprise-scale scanning
  2. Or provision Azure VM Scale Sets as self-hosted runners with auto-scaling
  3. Pre-install CodeQL CLI bundle on the runner image for faster cold starts
  4. Use runs-on: [self-hosted, codeql] label to route CodeQL jobs to dedicated runners
  5. Configure runner groups to restrict which repos can use which runners (org-level control)

Custom CodeQL Queries

For enterprise-specific security requirements:

  • Custom query packs: Write QL queries for your org's specific patterns (e.g., "our internal auth library must always be called before database access")
  • Query suites: Bundle queries into suites and reference them in your CodeQL config: queries: my-org/security-queries
  • Model packs: Teach CodeQL about your custom frameworks — define what methods are "sources" (user input) and "sinks" (dangerous ops) in your internal libraries
  • Third-party integrations: CodeQL results can be exported as SARIF and imported into any SIEM or GRC tool (Splunk, ServiceNow, Archer)

Key talking point: "CodeQL doesn't just find bugs — it understands your code the way a senior security engineer would. It follows the data from user input to database query and says 'this path is exploitable.' And then Copilot writes the fix. That's not a scanner — that's an AI security team member."

Supply Chain Security — Why It Matters

Supply chain attacks are the #1 growing threat vector. SolarWinds, Log4Shell, the XZ Utils backdoor — all supply chain. The average enterprise application has 200-500 dependencies, and each of those has its own dependencies. One vulnerable package deep in the tree can compromise everything.

Dependency Graph — What to Show

When you open Insights → Dependency Graph, walk through:

  • Dependencies tab: Shows every package your project uses, organized by manifest file. Point out the counts — "Your project has X direct dependencies and Y transitive dependencies."
  • Dependents tab: Shows who depends on YOUR code (important for internal library maintainers).
  • SBOM export: Click "Export SBOM" to download an SPDX-format Software Bill of Materials. "This is what your compliance team needs for Executive Order 14028, SOC 2, or any federal contract."

Out-of-the-box detection: The graph parses these manifest files automatically — no config needed:

  • npm: package.json, package-lock.json
  • Python: requirements.txt, Pipfile, setup.py, pyproject.toml
  • Maven: pom.xml
  • Gradle: build.gradle
  • NuGet: .csproj, packages.config
  • Go: go.mod, go.sum
  • Ruby: Gemfile, Gemfile.lock
  • Rust: Cargo.toml, Cargo.lock
  • PHP: composer.json
  • Dart: pubspec.yaml
  • Elixir: mix.exs
  • GitHub Actions: workflow YAML files

Custom/private libraries: If a customer asks "what about our internal packages?" — three answers:

  1. If listed in manifests: They appear automatically (e.g., @mycompany/auth-lib in package.json)
  2. Dependency Submission API: For custom build systems (Bazel, Make, proprietary), use the API to submit dependency snapshots after your build runs
  3. Private advisories: Create security advisories for your internal packages so Dependabot alerts fire on your own CVEs

Dependabot — The Three Modes

  1. Dependabot Alerts: Notifies you when a dependency has a known CVE. You see the CVE, severity (CVSS), affected versions, and patched version. This is passive — it tells you, you decide what to do.
  2. Dependabot Security Updates: ACTIVE mode — automatically creates PRs to bump the vulnerable dependency to the nearest patched version. The PR includes changelog, compatibility notes, and CI check results.
  3. Dependabot Version Updates: Keeps ALL dependencies current (not just vulnerable ones). Configure via .github/dependabot.yml to run daily/weekly/monthly. Reduces the "update debt" that makes emergency patches painful.

How DNS Tunneling Works
The attacker encodes stolen data as subdomain queries: instead of looking up google.com, they look up c3RvbGVuX2RhdGE.evil-domain.com — that random subdomain IS the stolen data, base64-encoded. The agent detects this by measuring entropy.

🎤 Presenter Notes
DNS tunneling is one of the sneakiest exfiltration methods because DNS traffic is almost never blocked — every network needs DNS. The attacker encodes stolen data as subdomain queries: instead of looking up "google.com", they look up "c3RvbGVuX2RhdGE.evil-domain.com" — that random subdomain IS the stolen data, base64-encoded. The agent detects this by measuring entropy — random strings have high entropy, legitimate domains don't. Simple math, devastating effectiveness.
